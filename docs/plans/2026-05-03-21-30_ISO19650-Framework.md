# ISO 19650 Framework + Health-Score Retirement

**Track:** Backbone (precedes embed track from
`2026-05-03-21-15_Forward-Deployed-Embed.md` and propagates platform-wide)
**Status:** Plan
**Authors:** omarchy session, draft for cross-machine review with edkjo

---

## Goal

Build the platform's logic explicitly around **ISO 19650** (the international
BIM information management standard) and the **IFC schema**. The dashboard
metric becomes **requirement fulfillment** ("4 of 13 EIRs fulfilled" with
drill-in), not an abstracted health score.

Concrete outcomes:

1. Domain models for the ISO 19650 information requirements taxonomy:
   `InformationRequirement`, `RequirementFulfillment`, `BEPSection`, with a
   clear mapping to existing `ProjectConfig` verification rules.
2. The verification engine reframed: rules become **acceptance criteria** on
   requirements, and the engine emits `RequirementFulfillment` rows instead
   of (or alongside, briefly) `health_score`.
3. `health_score` is retired across the API, the frontend, and the
   `dashboard-metrics` endpoint.
4. Quality issues (untyped, orphan, empty container, etc. — see
   `feedback-bad-models-are-the-product.md`) are surfaced as **gaps against
   specific requirements**, not as standalone scores.

This work is the prerequisite for the Requirements Fulfillment dashboard
(MVP #1 in the embed track) and reframes every existing dashboard surface in
the app over time.

---

## Why now

Three forcing functions:

1. **Embed track depends on it.** The first MVP dashboard in the
   forward-deployed embed track is "Requirements Fulfillment" — that
   dashboard cannot ship without the domain models below.
2. **Skiplum-pages already proved the framing.** The Skiplum dashboards use
   "X of Y EIRs fulfilled" and operators understand it immediately.
   `health_score` is the abstraction that gets in their way.
3. **Existing `health_score` is shallow** — it's `passed / checked * 100` per
   project/model/discipline. It hides which requirement a failure belongs
   to, who owns the requirement, and what the contractual evidence is. That
   metadata is exactly what ISO 19650 contributes.

## Reference: ISO 19650 vocabulary (only what's load-bearing for us)

| Acronym | Term | What it is | Owner |
|---|---|---|---|
| **EIR** | Exchange Information Requirements | Client's information requirements at exchange points (handover, milestones). Contractual. | Client / asset owner |
| **OIR** | Organisational Information Requirements | What the organization needs for asset management. Long-lived. | Organization |
| **AIR** | Asset Information Requirements | What's needed for the asset's lifecycle. | Asset operator |
| **PIR** | Project Information Requirements | What's needed for the project. Project-scoped. | Project lead |
| **BEP** | BIM Execution Plan | The supplier's response to the EIR — how info will be delivered. | Lead designer / contractor |
| **MIDP** | Master Information Delivery Plan | Schedule of all project info deliverables. | Lead designer |
| **TIDP** | Task Information Delivery Plan | Task-level slice of the MIDP. | Task team |
| **CDE** | Common Data Environment | Where everything lives. (Sprucelab is one — this is context, not a model.) | Platform |

**Sprucelab's responsibility:** the entire information delivery cycle —
ordering, contracting, delivering, verifying. Not just the verification end.
The platform helps users **formally order their data needs**, turns those
needs into structured **EIRs**, accepts a structured **BEP** as the
supplier's response, and then verifies delivered IFC + supporting data
against that contract. MIDP/TIDP scheduling layers on top.

Most of the industry does this loop in PDF/Excel/Word documents that are
hard to update, harder to verify against, and impossible to query. Sprucelab
makes every step structured and machine-checkable.

**IDS is the machine-readable carrier.** The `InformationRequirement` data
model maps to **IDS** (buildingSMART's open XML standard for IFC information
requirements — see `ids-as-interop-format` memory). Sprucelab is
bidirectional on IDS:

- **Generate** — IDS export per project, per use case, or per EIR. The
  supplier (or any IDS-aware tool: Solibri, ACCA usBIM, IfcOpenShell,
  BlenderBIM) can run the file through their own validator. Sprucelab is
  the authoring tool; IDS is the distribution format.
- **Import** — clients who already express their EIRs in IDS (or who use
  another tool that exports it) drop the file in; Sprucelab populates
  `InformationRequirement` rows directly. Round-trip semantics: import →
  no changes → export → byte-equivalent (modulo whitespace).

ISO 19650 is the contractual envelope (who signs, when, against which
gate); IDS is what's inside the envelope (the machine-checkable
specification). Sprucelab does both natively.

## The information delivery cycle (the user journey)

This is the flow the platform formalizes. Each step is a structured
artifact in Sprucelab, not a document attachment.

```
1. User picks a use case          ┐
   "I need LCA-readiness"          │  Plain-language entry. UseCaseTemplate
   "We're going to clash-check"    │  primes a default requirement bundle.
                                   ┘

2. User orders data needs          ┐
   "Material layers per element"   │  Structured form, one row per data
   "EPD reference in pset_X"       │  need. Templates auto-populate; user
   "NS3451 classification on all"  │  edits, adds, removes. Plain-language
                                   ┘  side, structured-data side.

3. EIR drafted                    ┐
   Formal ISO-19650-shaped EIR    │  Sprucelab translates ordered needs
   sections, codes, owners,       │  into formal EIR rows. Acceptance
   acceptance criteria.           │  criteria are derived (and editable).
                                  ┘

4. EIR signed off                 ┐
   Status: draft → review → signed│  Contractual. Versioned. Audit trail.
   Client + supplier both sign.    ┘

5. BEP drafted                    ┐
   Supplier responds: per-EIR,    │  Structured response: per requirement,
   which deliverables, when, in   │  which deliverables (IFC views, COBie,
   which format, by whom.         │  Excel), at which gate, in which format,
                                  ┘  by which task team. Templates per
                                     deliverable type.

6. BEP signed off                 ┐
   Status: draft → review → signed│  Contractual basis for verification.
                                  ┘

7. IFC + supporting data delivered┐
   Existing upload + extraction.  │  Lands in extraction → claims → ready
                                  ┘  to be evaluated against the signed BEP.

8. Verification + fulfillment     ┐
   Engine evaluates each deliverable│ RequirementFulfillment rows written.
   against its BEP-defined criteria,│ Use-case readiness lights flip green.
   which evaluate against the EIR.  │ Gate status updates.
                                    ┘  Operators see what's still missing.
```

Sprucelab participates at every step. Steps 1–6 are the **authoring** half
(new for v2 — most of the industry skips this and goes straight to step 7
on a verbal agreement). Steps 7–8 are the **verification** half (existing
strength — the verification engine, claims, ProjectConfig rules already
work here, just need to be wired through requirements + BEP).

The use case is the **entry point**, not just a projection. Users land on
"pick what you're trying to do" — LCA, clash, cost, handover — and the
platform walks them through ordering the requirements that workflow needs.

## The three axes: ownership × purpose × time

A single information requirement lives at one point in three orthogonal
spaces. Conflating them is the trap; modelling them separately is what
makes the dashboards work.

| Axis | What it answers | Vocabulary |
|---|---|---|
| **Ownership** (ISO 19650) | Who needs this info, for what lifecycle? | EIR, OIR, AIR, PIR, BEP-section |
| **Purpose** (use case / scope) | What workflow does this info enable? | LCA-ready, Clash-ready, Cost-ready, Energy-ready, Handover-ready, Schedule-ready |
| **Time** (gate / milestone) | When must this info exist? | Concept, Preliminary, Detailed, Construction, As-built, Operation |

Examples:

- *"Every wall has a material layer with EPD reference"* — this is **an EIR**
  (client asked for it), part of the **LCA-ready** use case, with target gate
  **Preliminary design**.
- *"Geometry is federated and clash-checkable"* — this is **a PIR** (project
  lead's working requirement), part of the **Clash-ready** use case, with
  target gate **Detailed design**.
- *"Every space has an area and a function code"* — this is **an AIR** (asset
  ops will use it), part of the **Handover-ready** and **Energy-ready** use
  cases, with target gate **As-built**.

The dashboards present these axes orthogonally:

- **By ownership** — "of the 13 EIRs the client signed off on, 4 are
  fulfilled" (the original Skiplum-pages framing).
- **By use case** — "LCA-ready: ☑️, Clash-ready: ⚠️, Cost-ready: ❌"
  (traffic-light tiles, each a click-through to the requirements that drive
  the status).
- **By gate** — "We're entering Detailed Design. Of the 17 requirements
  due by this gate, 14 are fulfilled, 2 are partial, 1 is unfulfilled."

All three are projections over the same underlying `InformationRequirement` +
`RequirementFulfillment` rows. No duplicated data — the views are derived.

### Use cases (purpose axis)

A `UseCase` is a named bundle of requirements that together enable some
downstream activity. Initial set (refine in implementation):

- **LCA-ready** — material layers, EPD references, quantities, NS3451
  classification, structural roles. Powers `/api/types/export-reduzer/`,
  `export-oneclicklca/`.
- **Clash-ready** — federated geometry, complete discipline coverage,
  consistent CRS, no orphan elements above N% threshold.
- **Cost-ready** — quantities, type-level cost units, structural roles.
- **Energy-ready** — spaces with areas + function codes, building envelope
  identified, HVAC zones, U-values per assembly.
- **Handover-ready** — asset register complete, AIR fulfillment, COBie-shaped
  data, document references attached.
- **Schedule-ready** — task team assignments, MIDP/TIDP linkage, milestone
  flags.

These are **first-class dashboards** with traffic lights. The user lands on
the project, sees six tiles: ☑️ LCA-ready, ⚠️ Clash-ready, ❌ Cost-ready,
… Click any tile → that use case's requirement list with drill-in.

### Gates (time axis)

A `ProjectStage` (gate) defines when in the project lifecycle a requirement
must be fulfilled. The schedule is project-configurable but maps to common
frameworks (Norwegian skisse → forprosjekt → detaljprosjekt → utførelse →
FDV; UK RIBA stages 0–7; ISO 19650 information delivery cycle gateways).

A requirement has a `target_gate` — the stage by which it must be fulfilled.
Status against a gate:

- **On track** — requirement fulfilled, current stage ≤ target gate.
- **At risk** — requirement partial, current stage = target gate − 1.
- **Overdue** — requirement unfulfilled, current stage ≥ target gate.

The "Where are we for the next gate?" tile is a stack of these statuses.

### New models

```python
# apps/requirements/models.py (new app)

class InformationRequirement(models.Model):
    """
    One requirement, ISO-19650-typed. Owned by a project, optionally
    grouped by section (e.g. "EIR §4.2 — Geometry"), referencing a
    template (canonical EIR/OIR libraries can live as templates).
    """

    class Type(models.TextChoices):
        EIR = "EIR", "Exchange Information Requirement"
        OIR = "OIR", "Organisational Information Requirement"
        AIR = "AIR", "Asset Information Requirement"
        PIR = "PIR", "Project Information Requirement"
        BEP_SECTION = "BEP", "BEP Response Section"

    project = models.ForeignKey("projects.Project", on_delete=models.CASCADE)
    type = models.CharField(max_length=8, choices=Type.choices)  # ownership axis
    code = models.CharField(max_length=64)        # "EIR-07", "OIR-03"
    name = models.CharField(max_length=255)
    # Plain-language data need (the "ordered" form, before formalization).
    plain_language_need = models.TextField(blank=True)
    description = models.TextField(blank=True)
    owner_role = models.CharField(max_length=128, blank=True)
    section = models.CharField(max_length=128, blank=True)
    # Acceptance criteria as structured rules (the bridge to the existing
    # verification engine). Shape mirrors ProjectConfig.rules entries.
    acceptance_criteria = models.JSONField(default=list)
    source_document_ref = models.CharField(max_length=512, blank=True)

    # Purpose axis — which use cases does this requirement enable?
    use_cases = models.ManyToManyField(
        "UseCase", related_name="requirements", blank=True
    )
    # Time axis — by which gate must this be fulfilled?
    target_gate = models.ForeignKey(
        "ProjectStage", null=True, blank=True, on_delete=models.SET_NULL,
        related_name="requirements_due"
    )

    # Workflow — requirements are contractual artifacts.
    class Status(models.TextChoices):
        DRAFT = "draft"
        ORDERED = "ordered"        # plain-language stage, not formalized
        IN_REVIEW = "in_review"
        SIGNED = "signed"          # contractual; changes require versioning
        SUPERSEDED = "superseded"

    status = models.CharField(
        max_length=16, choices=Status.choices, default=Status.DRAFT
    )
    version = models.IntegerField(default=1)
    signed_at = models.DateTimeField(null=True, blank=True)
    signed_by = models.ForeignKey(
        "auth.User", null=True, blank=True, on_delete=models.SET_NULL,
        related_name="signed_requirements"
    )


class UseCaseTemplate(models.Model):
    """
    Canonical, library-shipped use case (LCA-ready, Clash-ready, etc.)
    with a default requirement skeleton. Users adopt a template into a
    project, customize, and the customized version becomes the project's
    UseCase + draft EIR. Not project-scoped.
    """
    code = models.CharField(max_length=64, unique=True)  # "lca-ready"
    name = models.CharField(max_length=128)              # "LCA Ready"
    description = models.TextField()
    # Skeleton requirements: list of structured "data need" rows the user
    # can adopt. Each row turns into an InformationRequirement on adoption.
    skeleton = models.JSONField(default=list)


class UseCase(models.Model):
    """
    A bundle of requirements that together enable a downstream activity
    (LCA, clash control, cost estimation, etc.). Project-scoped, may be
    derived from a UseCaseTemplate or custom-built.
    """
    project = models.ForeignKey("projects.Project", on_delete=models.CASCADE)
    template = models.ForeignKey(
        UseCaseTemplate, null=True, blank=True, on_delete=models.SET_NULL
    )
    code = models.CharField(max_length=64)   # "lca-ready", "clash-ready"
    name = models.CharField(max_length=128)  # "LCA Ready"
    description = models.TextField(blank=True)
    # Display order on the dashboard. Lower = more prominent.
    sort_order = models.IntegerField(default=0)
    # Threshold for "ready" — fulfilled / total ratio, default 1.0 (all of them).
    ready_threshold = models.FloatField(default=1.0)


class Deliverable(models.Model):
    """
    The supplier's commitment to deliver a specific information slice in
    response to one or more requirements. Lives under a BEPSection. The
    verification engine evaluates IFC + supporting data against the
    deliverable's acceptance criteria (refining the parent requirement's
    criteria with concrete format/timing constraints).
    """

    class Format(models.TextChoices):
        IFC = "ifc", "IFC"
        COBIE = "cobie", "COBie"
        EXCEL = "excel", "Excel"
        PDF = "pdf", "PDF"
        OTHER = "other", "Other"

    bep_section = models.ForeignKey(
        "BEPSection", related_name="deliverables", on_delete=models.CASCADE
    )
    fulfills = models.ManyToManyField(
        InformationRequirement, related_name="deliverables"
    )
    code = models.CharField(max_length=64)    # "DEL-LCA-MAT"
    name = models.CharField(max_length=255)
    format = models.CharField(max_length=16, choices=Format.choices)
    target_gate = models.ForeignKey(
        "ProjectStage", null=True, blank=True, on_delete=models.SET_NULL
    )
    task_team = models.CharField(max_length=128, blank=True)
    # Concrete acceptance criteria — refines the parent requirement's
    # criteria (e.g. EIR says "material layers must exist", deliverable
    # says "as IfcMaterialLayerSet in IFC4 ReferenceView, by Preliminary
    # Design").
    acceptance_criteria = models.JSONField(default=list)


class ProjectStage(models.Model):
    """
    Gate / milestone in the project lifecycle. Project-scoped so each
    project can use its own framework (Norwegian skisse..FDV, RIBA 0..7,
    ISO 19650 gateways, custom).
    """
    project = models.ForeignKey("projects.Project", on_delete=models.CASCADE)
    code = models.CharField(max_length=64)   # "preliminary", "ribastage4"
    name = models.CharField(max_length=128)  # "Preliminary Design"
    sort_order = models.IntegerField()
    is_current = models.BooleanField(default=False)  # one row True per project
    target_date = models.DateField(null=True, blank=True)


class RequirementFulfillment(models.Model):
    """
    One evaluation of one requirement, against a project state at a point
    in time. The verification engine writes these.
    """

    class Status(models.TextChoices):
        FULFILLED = "fulfilled"
        PARTIAL = "partial"
        UNFULFILLED = "unfulfilled"
        NOT_EVALUATED = "not_evaluated"

    requirement = models.ForeignKey(InformationRequirement, on_delete=models.CASCADE)
    project = models.ForeignKey("projects.Project", on_delete=models.CASCADE)
    evaluated_at = models.DateTimeField(auto_now_add=True)
    status = models.CharField(max_length=16, choices=Status.choices)
    fulfilled_count = models.IntegerField(default=0)
    total_count = models.IntegerField(default=0)
    # Structured gap data — what's missing, why. Drives the dashboard drill-in.
    gaps = models.JSONField(default=list)
    # Pointer to the ExtractionRun / ModelVersion this evaluation reflects
    extraction_run = models.ForeignKey(
        "models.ExtractionRun", null=True, on_delete=models.SET_NULL
    )


class BEPSection(models.Model):
    """
    A section of the supplier's BEP — the contractual response to one or
    more EIRs. v1: ingested as project config; future: authored in-platform.
    """
    project = models.ForeignKey("projects.Project", on_delete=models.CASCADE)
    code = models.CharField(max_length=64)
    title = models.CharField(max_length=255)
    body = models.TextField()
    responds_to = models.ManyToManyField(
        InformationRequirement, related_name="bep_responses"
    )
```

Indexes (initial — refine in PR):
- `(project, type, code)` unique on `InformationRequirement`
- `(requirement, evaluated_at desc)` on `RequirementFulfillment`
- `(project, status, evaluated_at desc)` on `RequirementFulfillment` for

### Verification engine reframe

`apps/entities/services/verification_engine.py` currently produces:

```python
@dataclass
class VerificationResult:
    health_score: float
    issues: list
    per_type_status: dict
    ...
```

The new shape:

```python
@dataclass
class VerificationResult:
    fulfillments: list[RequirementFulfillmentDTO]
    # Aggregate counts derivable from fulfillments — no top-level score
    summary: RequirementSummary  # {fulfilled, partial, unfulfilled, total}
```

Each `RequirementFulfillmentDTO` ties to one `InformationRequirement`. The
engine groups acceptance criteria by their parent requirement, evaluates
each, and emits one fulfillment row per requirement.

**Existing rules in `ProjectConfig` migrate to acceptance criteria** on
synthesized "ungrouped" requirements during the cutover, then are
reorganized into proper EIRs as projects formalize their information
requirements.

## Use-case status: traffic-light derivation

A use case's status is computed from the fulfillments of its bundled
requirements. The default rule:

```
ratio = fulfilled_count / total_count   # over the use case's requirements
if ratio >= use_case.ready_threshold:    status = "ready"        (green)
elif ratio >= 0.5:                       status = "in_progress"  (yellow)
else:                                    status = "not_ready"    (red)
```

Override per use case if needed (e.g. clash-ready might need 100% on
"federation complete" but tolerate < 100% on edge requirements).

The viewer crosswire: clicking the **LCA-ready** tile sets
`filter_context.use_case_id = <id>` — every other tile re-projects to that
use case's data, the viewer isolates the elements that contribute to it.
Dashboard cross-filter doesn't care that "use case" is a different axis;
the filter context just carries it.

### Gate status: per-requirement and aggregate

For a single requirement r at gate g (the project's current stage):

```
if r.target_gate is None:                           "no_gate"
elif r.target_gate.sort_order > g.sort_order:       "future"      (not yet due)
elif r.target_gate.sort_order == g.sort_order:
    if r.fulfillment.status == "fulfilled":         "on_track"
    elif r.fulfillment.status == "partial":         "at_risk"
    else:                                           "overdue"
else:                                               "overdue"
```

Aggregated to project: counts of each status. Tile: "Detailed Design gate
in 12 days. 14 on track / 2 at risk / 1 overdue."

## Out of scope (for this plan)

- **AI-assisted plain-to-formal translation** — when a user enters a plain
  data need ("I need every wall to have an EPD"), the platform should
  eventually suggest the structured acceptance criteria. v1 is template +
  manual; LLM-assisted comes later (and aligns with the deferred
  Sprint 6.3 LLM extraction work).
- **MIDP/TIDP scheduling** — Deliverable + ProjectStage gives us the raw
  pieces. A full delivery-plan UI (Gantt-shaped) is later.
- **Document extraction** — automatically pulling EIRs from PDF contracts
  is later. For v1, requirements come from use-case templates +
  structured input + Excel import.
- **Field/regulatory compliance merge** — the `apps/field/` checklists
  remain separate. They serve adjacent but distinct workflows.
- **Inter-project requirement reuse** — OIR/AIR conceptually span
  projects; v1 keeps them per-project.

3. **Recovering `apps/bep/` from archive** — what's salvageable, what's
   replaced? Spelunk archive before designing `BEPSection` from scratch.

4. **Requirements seeding source** — Excel template, JSON, in-app form? For
   v1 pick one (probably Excel template that ports the rule format we
   already use for type classification — `GET /api/requirements/export-excel/`,
   `POST /api/requirements/import-excel/`).

5. **Multi-project requirements** — OIR/AIR conceptually span projects.
   v1: scope every requirement to a single project; v2: introduce
   organization-level scopes.

6. **Versioning** — when an EIR changes mid-project, do we keep the old
   one and version it, or mutate? Lean: mutate with audit trail (per
   Agent-Workflows roadmap #3).

7. **Standard EIR libraries** — are there public EIR templates (UK BIM
   Framework, ISO sample EIRs, Norwegian Statsbygg standards) we should
   ship as starter templates? Worth a 1-hour research spike before the
   Phase 1 PR.

8. **Skiplum-pages reuse** — Skiplum-pages already renders the "X of Y
   fulfilled" framing. What does the data shape look like there? If we
   match it, embedding into skiplum-pages becomes much easier (the data
   already speaks the same language). Audit before finalizing the
   `RequirementFulfillment` shape.

9. **Use-case starter set** — LCA-ready, Clash-ready, Cost-ready,
   Energy-ready, Handover-ready, Schedule-ready. Which ship at MVP,
   which wait? LCA + Clash + Handover are the obvious starting trio
   (LCA leverages existing `export-reduzer`/`export-oneclicklca`
   plumbing; clash matches existing federation work; handover is what
   the asset side cares about). Confirm before PR 2.

10. **Gate framework defaults** — does the platform ship with a default
   stage list (Norwegian skisse..FDV) or stay framework-agnostic?
   Lean: ship Norwegian as default (Skiplum context) with RIBA + ISO
   19650 gateways as importable templates. Project picks one at setup.

11. **Composition of axes** — is the dashboard's primary entry the
   ownership view (EIR list), the purpose view (use-case tiles), or
   the time view (gate status)? All three exist; one has to be the
   landing surface. Lean: **purpose view as landing** (six traffic-light
   tiles answers "where are we?" fastest), with one click to drill into
   ownership or gate detail.

12. **Ordering UX shape** — step 2 of the delivery cycle ("user formally
   orders their data needs") is a new authoring surface. Form-based
   row-by-row entry with template auto-populate? Conversational?
   Spreadsheet-like? Lean: form-based with template auto-populate for
   v1; conversational is a later AI track.

13. **Sign-off mechanics** — multi-party sign-off (client + supplier)
   for an EIR; supplier-only sign-off for a BEP? Or single-party for
   v1 with multi-party as a flag? Lean: single-party for v1, structured
   audit trail (who signed, when), multi-party as a v2 enhancement.

14. **Versioning when a signed requirement changes** — clone with
   incremented `version` and `status=superseded` on the old? Or in-place
   mutate with audit log? Lean: clone-and-supersede — preserves
   historical fulfillments cleanly.

15. **Use case templates as a community/library asset** — do we ship
   public, vetted templates (Skiplum / industry / open) that projects
   can adopt? Or is the template library project-private? Lean: ship
   a small starter library (Skiplum-authored), let projects clone +
   customize, leave a public-template-marketplace for v2. Templates
   should themselves be exportable as IDS bundles.

16. **IDS schema mapping** — exact mapping between
   `InformationRequirement.acceptance_criteria` JSON and IDS
   `<applicability>` + `<requirements>` blocks. This deserves its
   own short design note alongside PR 5a. Blocker: any criterion
   shape we can't express in IDS is a smell — either we add the
   IDS-equivalent or we accept it's Sprucelab-private (and document
   why).

17. **IDS validator vs internal engine** — short-term we run both
   (engine for IDS-expressible stuff via IfcOpenShell, engine for
   things IDS can't express like cross-element relationships, MIDP
   timing, federation completeness). Long-term, can we push more
   into IDS? Or do we extend IDS via vendor-specific blocks (the
   spec allows this)? Decision deferred until PR 5b lands and we
   have real data on what doesn't fit.

## Success criteria

- `health_score` does not appear in any code path (Phase 4 done).
- **IDS round-trip works**: import a buildingSMART sample IDS file, make
  zero edits, export — output is byte-equivalent to input (modulo
  canonical whitespace).
- **IDS export validates against the buildingSMART schema** (XSD) for
  every project's requirement set.
- **IDS export from Sprucelab + IDS validation in IfcOpenShell** produces
  the same fulfillment counts as Sprucelab's internal verification engine
  on the same IFC. The two should converge — eventually the engine wraps
  the IfcOpenShell IDS validator directly.
- Requirements Fulfillment dashboard ships and shows real data for at least
  one project (G55 — the existing seeded project), framed three ways:
  - by ownership (EIR/OIR/AIR/PIR breakdown)
  - by use case (traffic-light tiles: LCA-ready, Clash-ready, Cost-ready,
    Handover-ready)
  - by gate (current-vs-next-stage status)
- A new project can be configured by importing an EIR list (Excel) plus a
  use-case bundle and a project-stage schedule, and the verification engine
  emits fulfillments + use-case statuses + gate readiness without code
  changes.
- Every quality issue (untyped, orphan, etc.) is attributable to at least
  one requirement in test fixtures, and to at least one use case.
- The embed track's Requirements Fulfillment dashboard renders the same
  framing as skiplum-pages does today (within reason — UI primitives may
  differ; the metric framing must match).
- Clicking the **Clash-ready** tile cross-filters every other tile (viewer
  included) to clash-ready-relevant data. Same for LCA-ready, etc. — the
  filter context carries `use_case_id` like any other dimension.

## Implementation order

1. `docs: ISO 19650 framework plan` — this doc.
2. `requirements: domain models + migrations` (Phase 1) — `apps/requirements`,
   `InformationRequirement` + `RequirementFulfillment` + `BEPSection` +
   `UseCase` + `ProjectStage`, data migration that creates a default
   `InformationRequirement` per existing rule.
3. `requirements: verification engine extension` — engine emits fulfillments
   + use-case statuses + gate readiness alongside health_score (all four
   for now during cutover).
4. `requirements: API surface` (Phase 2) — `/api/requirements/`,
   `/api/use-cases/`, `/api/project-stages/` viewsets;
   `dashboard-metrics` adds the `requirements`, `use_cases`, and `gate`
   blocks.
5. `requirements: Excel import/export` — seeding workflow for requirements
   + use-case bundles + project-stage schedule.
5a. `requirements: IDS import/export` — buildingSMART IDS XML, two-way.
   Round-trip test target. `apps/requirements/services/ids_io.py` plus
   endpoints `POST /api/requirements/import-ids/` and
   `GET /api/requirements/export-ids/?project=...&use_case=...`.
   Lands alongside Excel — IDS is the open-standard interop, Excel is
   the operator-friendly seeding path. Same data, two carriers.
5b. `requirements: IDS-validator integration` — wrap the IfcOpenShell IDS
   validator so the verification engine can run "Sprucelab requirements →
   IDS export → IfcOpenShell validate" as one path, and produce
   fulfillments from validator output. Eventually displaces the
   hand-rolled rule executor for IDS-expressible criteria.
6. `embed: Requirements Fulfillment dashboard` (from embed track) — first
   consumer of the new API. Includes use-case traffic-light tiles + gate
   status tile.
7. `frontend: migrate Type Browser to requirements framing` — Phase 3.
8. `frontend: migrate model card + project header` — Phase 3 cont.
9. `requirements: drop health_score` (Phase 4) — destructive cleanup.

PRs 2–4 land in one session ideally. PR 5 unblocks real-world testing. PRs
6–8 happen in parallel across the embed track and the in-app dashboards.
PR 9 is the final cleanup.
