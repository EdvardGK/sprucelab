# Session: Phase 1 PR 1.2 — Viewer migrated onto useProjectFilter

## Summary
Shipped PR 1.2 of the frontend refresh roadmap (Phase 1 — Filter
primitive). The federated viewer chrome now reads/writes the project-
scoped `ProjectFilterProvider` mounted in PR 1.1. Click semantics
flipped from toggle-hide (exclusion) to toggle-isolate (inclusion) per
capture-doc Item 014. The legacy Zustand `useViewerFilterStore` and the
parallel `?f=` URL channel are deleted; localStorage persist lands once
on the new provider as `sprucelab-filter-v3`.

Pushed direct to `main` (commit `91938b3`) per
`feedback-trunk-based-until-go-live.md`.

## Changes

- **Schema (`frontend/src/lib/embed/types.ts`)** — additive:
  `excluded_ifc_class?: string[]` (right-click "hide this class" target)
  and `hidden_models?: string[]` (federated platform-panel hide-eye
  target). Both threaded through `createFilterContext`. No protocol
  bump (matches PR 1.1's pattern; the schema's additive-only rule
  covers it).
- **Provider (`contexts/ProjectFilterProvider.tsx`)** — new dimensions
  added to `ArrayDimensionKey`; `setExcludedIfcClass` and
  `setHiddenModels` action creators.
- **URL hook (`hooks/useProjectFilterUrl.ts`)** — encodes/decodes the
  two new facets in `projectToPayload`.
- **Persist hook (`hooks/useProjectFilterPersist.ts`, NEW)** —
  localStorage `sprucelab-filter-v3`, `partialize`-omits invariants
  (`mode`, `project_id`, `protocol_version`, `selected_express_id`)
  AND ephemeral selection facets (`selected_type_ids`,
  `selected_global_ids`, `color_by`). One-shot wipe of legacy
  `sprucelab-viewer-filter-v2` on first mount.
- **Mount (`pages/ProjectShell.tsx`)** — persist hook runs *before* URL
  hook so `?d=` deeplinks win over localStorage on hydrate. Provider
  now keyed on `:id` so React unmounts the subtree on project switch
  (mirrors the old store's `setScope` reset behavior).
- **Derive util (`lib/filters/deriveTypeVisibility.ts`, NEW)** —
  inclusion (`ifc_class[]`) + subtractive exclusion
  (`excluded_ifc_class[]`) → `Record<string, boolean>` matching the
  prop shape `UnifiedBIMViewer` already accepts. The viewer itself is
  unchanged (already prop-driven).
- **`ViewerFilterPanel.tsx`** — Zustand selectors swapped for
  `useProjectFilter` + `useProjectFilterActions`. Click handler is
  now `handleIsolate(cls, additive)` — replace inclusion with
  `[cls]`, cmd/ctrl-click toggles membership in the existing set,
  click-on-only-active clears. Right-click adds/removes from
  `excluded_ifc_class[]`. Visual: chip neutral default,
  cyan-tinted bordered when in inclusion, rose strikethrough when
  excluded. "Hide all" button removed (no clean inclusion-model
  equivalent). "Show all" preserves floor + property facets.
  `floor_code` shape adapter — provider stores `string[]`, viewer
  reads `state.floor_code?.[0] ?? null`, writes `[code]` or
  `undefined`.
- **`FederatedViewer.tsx`** — same migration. Dropped `setScope`
  effect (project_id rides on the `<ProjectShell />` seed + the new
  `key={id}`), dropped `useViewerFilterUrl()` call, dropped manual
  `activeFilters` useState (chip list now derives entirely from
  context — split into `include-<cls>` / `exclude-<cls>` ids so
  `handleRemoveFilter` knows which facet to mutate).
- **Deleted** (via `gio trash`):
  `frontend/src/stores/useViewerFilterStore.ts`,
  `frontend/src/hooks/useViewerFilterUrl.ts`. The `?f=` URL channel
  is gone — pre-migration deeplinks won't round-trip. Acceptable per
  trunk-only / internal-only window.
- **Comment cleanup** — stale references to the now-deleted store in
  `ProjectFilterProvider` and `useProjectFilterUrl` removed.

## Inclusion-model semantics (locked)

For future readers / next PR:

- **Empty `ifc_class[]`** ⇒ all classes pass inclusion.
- **Non-empty `ifc_class[]`** ⇒ only those classes pass inclusion.
- **`excluded_ifc_class[]`** is subtracted *after* inclusion, so
  right-click "hide this class" works whether or not an inclusion
  filter is active.
- Visible iff `(includedSet === null || includedSet.has(cls)) &&
  !excludedSet.has(cls)`.

This is the contract every dashboard tile that touches IFC class
filtering will project from going forward. PR 1.3 makes
`CanvasStatusPanel` chips interactive against this same store.
