# Session: Phase 2.5 — drop the ProcessingReport compat shim

## Summary

Killed the legacy `ProcessingReport` surface end-to-end. The frontend Dev page
now reads directly from `/api/files/extractions/` (the flat `ExtractionRun`
endpoint). The Django shim at `/api/types/processing-reports/` is gone, the
`processing_reports` table is dropped via a new migration, and FastAPI's dead
full-pipeline `process_model` (along with `_create_report` and the repo-level
`create_processing_report`) is removed. 6/6 backend tests + frontend tsc are
green.

## Technical Details

The ordering matters: dropping the FastAPI write paths *before* dropping the
table prevents a transient window where the orchestrator could try to
`INSERT INTO processing_reports` against a table that no longer exists. Since
nothing was writing or reading the table after the orchestrator + Django
changes, the migration runs as a clean `DeleteModel`.

`_create_report` and `process_model` were dead in production already
(`process_model_types_only` is the only path the API endpoints call), so
removing them was strictly cleanup. Their bulk-insert dependencies in the
repo (`bulk_insert_entities`, `bulk_insert_properties`, etc.) and the
`EntityData/PropertyData/SpatialData` dataclasses are now unused but were
left in place — they're a separate dead-code sweep, and removing them risks
hitting an undiscovered consumer.

The frontend hook keeps the file name `use-processing-reports.ts` per the
plan, but its public exports are renamed (`useExtractions`,
`useExtraction`). To get the table view to show "Source File / Project /
Format / Size" without N+1s, the hook does one extra `GET /files/` and
joins client-side; this is fine for the dev page but not what we'd ship to
end users at scale (they should use a backend join when there's a real
table).

## Notes

- The Sidebar still links to `/dev/processing-reports` and the i18n key
  `nav.processingReports` is still "Processing Reports". The URL is
  user-facing dev tooling and was kept stable; rename when the user prefers.

  `ProcessingReportDetail.tsx`) were also kept for the same reason — the

  was rebuilt around real fields, none of that fakery had to be ported.
