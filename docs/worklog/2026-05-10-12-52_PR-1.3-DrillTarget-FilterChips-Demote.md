## Technical Details

**Decisions captured during planning (AskUserQuestion):**
- **Demote scope = ProjectDashboard only.** ModelWorkspace.tsx already
  cross-filters its embedded viewer via local React state
  (`viewerTypeVisibility`, `viewerStoreyFilter`); migrating that to
  `useProjectFilter` would conflict with how its `<UnifiedBIMViewer>`
  instance is configured and is bigger surgery — deferred to PR 1.3b
  or Phase 3 (Type Page v2).
- **Chip body click = isolate (PowerBI-style).** Click `+Wall` body
  with `+Wall, +Floor` active → state becomes `[+Wall]`. Other
  dimensions (excludes, floor) untouched. The X still removes a
  single chip.

**Why CanvasOverlays now reads `useProjectFilter` directly:** the chip
divider in CanvasStatusPanel is conditional on `hasFilters`. Previously
that signal came in via the `filters` prop. Pushing the `useProjectFilter`
call into the component is cleaner than threading a `hasFilters` boolean
through FederatedViewer purely for layout.

**Why isolate is a no-op on single-chip dimensions:** prevents redundant
dispatches that would still trigger a reducer pass + React re-render +
URL sync. `FilterChips.isolate()` returns early when the chip is already
the only entry in its dimension. The DrillTarget's `active` prop is set
in that case so the chip shows the active outline as a visual hint.

**Discipline drill case removed cleanly:** the `{type: 'discipline'; code}`
variant was the only path that opened a per-discipline filtered modal.
Post-demote, the discipline filter is achieved by clicking the bar
(cross-filter), and the per-discipline modal escape goes through the
DashCard's `Table2` icon which opens the `{type: 'models'}` drill (the
all-models table). If users later want a "view this discipline's models
as a table" affordance, it should be filter-aware in `drillConfig` —
out of scope here.
