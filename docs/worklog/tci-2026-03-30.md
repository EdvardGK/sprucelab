## 22:25 — [knowledge] TCI ↔ CI worklog integration
The auto-checkpoint hook (`write-worklog.py`) already tracks TCI file writes (via track-change.py PostToolUse hook) — so TCI edits show up in auto-commits and Notion. But the CONTENT of thoughts isn't merged.

Three integration levels identified:
1. **Convention only**: tci-YYYY-MM-DD.md is a companion file in docs/worklog/. CI reads it during /worklog to pick up context. No code changes.
2. **Hook enhancement**: Modify write-worklog.py to detect today's tci-* files and append a "TCI Captures" summary section to auto-checkpoint worklogs. ~20 lines of Python.
3. **Briefing integration**: Modify generate-briefing.py to also read latest tci-* files so next session's briefing includes TCI captures. ~10 lines.

Level 1 works now with zero changes. Levels 2-3 are incremental improvements.

## 22:30 — [decision] Single worklog, two authors
New idea: TCI creates the session worklog file at session start with a Notes/Thoughts section and starts filling it in. Then when CI runs /worklog, the skill reads the EXISTING file and adds the structured session content (Summary, Changes, Commits, Next) around the TCI's notes. One file, not two.

This is cleaner than tci-* companion files because:
- No merge step needed
- /worklog skill already writes to docs/worklog/ — it just needs to detect and append to an existing file instead of creating fresh
- The notes section provides context the CI wouldn't otherwise have
- Chronological thought stream + structured outcome in one place

Trade-off: file ownership is now shared (both instances write to it). Risk of mid-write conflicts if both instances edit simultaneously. Mitigation: TCI only appends to a `## Notes` section at the bottom, CI writes the top sections.

## 22:35 — [decision] TCI notes persist across multiple worklogs
Worklogs happen 2-4 times per hour (auto-checkpoints, compacts, manual /worklog). TCI's notes section shouldn't reset with each one — it's a running stream for the work session, not per-worklog.

This means TCI doesn't create one worklog file. Instead:
- TCI maintains a single **notes stream file** for the session (or a time block)
- Each /worklog auto-checkpoint creates its own file as usual
- The /worklog skill READS the TCI notes stream and includes relevant recent notes in its output
- The notes file outlives any individual worklog entry

This is actually back to the companion file model, but with the /worklog skill pulling from it rather than ignoring it. The TCI file is a running capture buffer. Worklogs are snapshots that reference it.

## 22:40 — [decision] Notes log = new worklog sidekick concept
Simpler framing: the notes log isn't a TCI-specific artifact. It's a new **type of worklog output** — a "notes log" that lives alongside session worklogs as a first-class citizen.

- Worklogs = structured snapshots (what was built, commits, next steps)
- Notes log = running thought stream (ideas, decisions, observations, questions)
- Both live in docs/worklog/, both are part of the session record
- The /worklog skill and hooks already know about docs/worklog/ — they just need to be aware that `notes-*.md` files exist and shouldn't be overwritten
- Any instance can append to the notes log — TCI does it by default, but a normal instance could too
- The notes log is the raw material; worklogs are the refined output

This decouples the concept from TCI entirely. TCI is just an instance role that happens to primarily write to the notes log. But the notes log is useful even without TCI — a coding instance could jot quick notes there too.

## 22:45 — [decision] PAUSE — rethinking scope
User pushback: the multi-instance cross-session coordination is getting too complicated. Real workflow is often separate instances working on separate parts of the code. The simpler the better.

Core need revisited: "I have thoughts while coding. I don't want to dump them into the coding instance's context." That's it. The thought capture instance just needs to be a scratchpad that writes to files. No cross-instance coordination protocol, no hook modifications, no worklog integration plumbing.

Maybe the answer is just: open a second Claude, tell it "you're my notes instance", and it writes to docs/. No protocol file, no config changes, no system integration. Just a convention.
