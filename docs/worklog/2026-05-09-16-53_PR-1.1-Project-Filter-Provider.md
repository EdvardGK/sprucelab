# Session: Phase 1 PR 1.1 — Project filter provider mounted

## Summary
Shipped PR 1.1 of the frontend refresh roadmap (Phase 1 — Filter primitive).
`DashboardFilterProvider` was renamed and extended into a project-scoped store
mounted once at a new `<ProjectShell />` parent route, with URL sync at
`?d=<base64>`. This unblocks PR 1.2 (ViewerFilterPanel migration onto the new
provider). No visible UI yet — purely foundational.

Branch: `frontend/pr1.1-project-filter-provider` (from `origin/main`).

## Changes

- **Schema (`frontend/src/lib/embed/types.ts`)** — additive extension to
  `FilterContext` (no protocol bump per the schema's own additive rules):
  `ns3451`, `verification`, `systems`, `selected_type_ids`,
  `selected_global_ids`, `color_by`. Moved `VerificationStatus` here as
  canonical home; added `ColorBy` interface. `createFilterContext` now
  forwards all new optional seed keys.
- **Provider (`contexts/ProjectFilterProvider.tsx`, replaces
  `DashboardFilterProvider.tsx`)** — renamed component + hooks
  (`useProjectFilter`, `useProjectFilterDispatch`,
  `useProjectFilterActions`). Reducer's `DimensionKey` extended for the new
  array-shaped facets; new `set_color_by` action handles the non-array
  dimension. Action creators added for every new facet.
- **URL sync (`hooks/useProjectFilterUrl.ts`, NEW)** — `?d=<base64>`,
  hydrate-once-on-mount + subscribe-to-changes pattern. Encodes everything
  except the four invariants (`mode`, `project_id`, `protocol_version`,
  `selected_express_id`). Coexists with the legacy `?f=` viewer-only
  channel.
- **Mount (`pages/ProjectShell.tsx`, NEW + `App.tsx` restructure)** — new
  layout component reads `:id` from `useParams`, wraps `<Outlet />` in
  `<ProjectFilterProvider />`, mounts `useProjectFilterUrl`. All 14
  `/projects/:id/*` routes converted to children of a single parent route,
  so the provider mounts once and survives intra-project navigation.
- **Back-compat (`stores/useViewerFilterStore.ts`)** — re-exports
  `VerificationStatus` from the schema home (deletion ships in PR 1.2 with
  the rest of the legacy store).
- **Doc tweak (`components/dashboard-primitives/README.md`)** — one-line
  reference updated.

## Technical Details

The router was a flat list with no parent layout; mounting the provider
properly required restructuring `/projects/:id/*` into a nested route under
`<ProjectShell />`. `withSuspense(guard(...))` collapses to the parent
wrapper since children inherit `RequireAuth` + `Suspense` from the
`<ProjectShell />` mount; per-route `lazy()` declarations stay so each
child still code-splits. The Embed surface (`/embed/:dashboard`)
intentionally bypasses the shell — embed manages filter state via its own
`useState` and shouldn't share the provider's reducer.

`DashboardFilterProvider` had **zero app consumers** at start of session
(only the file itself + the dashboard-primitives README), so the rename
was effectively cost-free. The legacy Zustand `useViewerFilterStore`
keeps its `sprucelab-viewer-filter-v2` persist key and `?f=` URL channel
through PR 1.1; the two stores deliberately coexist for one PR to keep
the migration atomic at the viewer in PR 1.2.

`selected_type_ids[]` and `selected_global_ids[]` are deliberately
distinct from `type_id[]` (filter facet) and `selected_express_id`
(single-element selection invariant). Selection-multi vs filter is the
inclusion-model distinction the capture doc Item 014 calls out — they
share the store but mean different things.
